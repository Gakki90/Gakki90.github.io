1. Spring 框架基础

Apache Maven:软件项目管理工具，管理项目的依赖，编译，文档等信息
groupId：组织的唯一id

artifactId：项目的唯一id

spring框架的四大原则：

  1. 使用POJO（Plain old java Object）进行轻量级和最小侵入式开发
  2. 通过依赖注入和基于接口编程实现松耦合
  3. 通过AOP和默认习惯进行声明式编程
  4. 使用AOP和模板减少模式化代码

1.1 IOC

控制翻转是通过依赖注入实现的，他的目的是解耦，所谓依赖注入是指使用Spring IOC容器（applicationContext）创建和维护对象间的依赖关系，而不是通过对象本身来负责。容器中的创建bean，在需要时，再将bean注入到类中

声明bean的注解：

  1. @Component ：没有明确的角色
  2. @Service：业务逻辑层
  3. Repository：数据访问层
  4. @Controller ：展现层

注入bean的注解：

  1. @Autowired：spring提供的注解
  2. @Inject
  3. @Resource

1.2 AOP

面向切面编程，目的也是为了解耦，将那些影响多个类的的公共行为封装到一个可重用模块中，将其命名为“aspect”方面，减少了重复代码，降低模块间的耦合性，有利于系统维护。

  1. @Aspect :声明这是一个切面
  2. @After ，@Before @Around 定义建言(advice)
  3. @PointCut :定义拦截规则，
  4. @JointPoint ：被拦截处称为连接点

@Bean的生命周期

  1. 实例化bean
  2. 设置对象属性（setter等
  3. 如果bean实现了BeanNameAware接口，工厂调用bean的setBeanName（）方法传入bean的ID
  4. 如果bean实现了BeanFactoryAware接口，工厂调用setBeanFactory（）传入工厂自身
  5. 将bean实例传递给bean的前置处理器的 postProcessBeforeInitialization(Object bean, String beanname)方法
  6. 调用bean的初始化方法，init（）
  7. 将bean传给bean的后置处理器 postProcessAfterInitialization(Object bean, String beanname)方法
  8. 使用bean
  9. 容器关闭前，调用bean的销毁方法

源码分析：

Bean的初始化就是在initializeBean方法中：initializeBean方法前面是populateBean：属性的赋值

  1. 先判断bean是否实现了aware接口,aware接口用于bean直接对ioc容器进行操作，有许多中接口，例如bean实现了ApplicationContextAware可以传入applicationContext容器

  2. 然后执行applyBeanPostProcessorsBeforeInitialization，前置处理器

  3. 然后执行invokeInitMethods，这是如果Bean实现了InitializingBean的接口，在这里就会处理其中的afterPropertiesSet方法，这个方法就是Bean的初始化方法，如果bean还指定了initMethod那么也会执行这个initMethod,最后完成Bean的初始化

  4. 最后执行applyBeanPostProcessorsAfterInitialization，后置处理器


protected Object initializeBean(String beanName, Object bean, @Nullable RootBeanDefinition mbd) {
        if (System.getSecurityManager() != null) {
            AccessController.doPrivileged(() -> {
                this.invokeAwareMethods(beanName, bean);
                return null;
            }, this.getAccessControlContext());
        } else {
            this.invokeAwareMethods(beanName, bean);
        }

        Object wrappedBean = bean;
        if (mbd == null || !mbd.isSynthetic()) {
            wrappedBean = this.applyBeanPostProcessorsBeforeInitialization(bean, beanName);
        }

        try {
            this.invokeInitMethods(beanName, wrappedBean, mbd);
        } catch (Throwable var6) {
            throw new BeanCreationException(mbd != null ? mbd.getResourceDescription() : null, beanName, "Invocation of init method failed", var6);
        }

        if (mbd == null || !mbd.isSynthetic()) {
            wrappedBean = this.applyBeanPostProcessorsAfterInitialization(wrappedBean, beanName);
        }

        return wrappedBean;
    }

@Bean的初始化和销毁方法

  1. 我们可以在@Bean注解上指定bean的初始化方法和销毁方法，注意，如果bean是单实例的，那么再容器创建时，bean就会被初始化，容器关闭时就会执行销毁方法，如果bean的多实例的，在使用时才会bean才会被初始化，容器关闭时不会执行销毁方法。
  2. 让bean实现 InitailizingBean接口以及DisposableBean接口，重写里面的方法，就等同于执行初始化方法和销毁方法
  3. 在bean中的方法上标注@PostConstruct就相当于指定了初始化方法，标注@PreDestory就相当于指定了销毁方法

@Value

获取配置文件中的某个属性的值

@PropertySource

导入配置文件 如：application.yml

@Autowired

自动装配：

  1. 默认先安装类型去容器中寻找组件，如果找到多个，那么再按照你定义的变量的名称去容器中查找
  2. @Qualifier（“xxx”）可以与@Autowired同时使用，指定你要找的组件的id
  3. @Autowired默认如果找不到会报错，使用@Autowired（requied=false）找不到不会报错
  4. @Primary使用在bean上，指定多个同类型的bean，优先使用有@Primary，但他的优先级低于@Qualifier

@Autowired可以装配到属性，方法，构造器，参数上

  1. 标注在方法上，那么方法的参数会从容器中获取，@Bean修饰的方法的方法参数就算不标注@Autowired也是自动从容器中获取
  2. 标注在构造器上，如果组件只有一个有参构造器，这个有参构造器的@Autowired可以省略。

1.4 AOP

1.4.1 AOP中的核心类

  • AnnotationAwareAspectJAutoProxyCreator ：继承了Spring ioc的扩展接口BeanPostProcessor（bean初始化前后的动作），主要用于获取advisor
  • DefaultPointcutAdvisor ：默认切面顾问，将切面增强设计（advice）与关注点（pointcut）结合起来的对象
  • Advice：通知，也就是AOP中增强的方法

1.4.2 流程：

大致分为三个步骤：

  1. 创建 AnnotationAwareAspectJAutoProxyCreator 对象

  2. 扫描容器中的切面，创建 PointcutAdvisor 对象

  3. 创建代理类，代理类的生成是在BeanPostProcessor接口的 postProcessAfterInitialization 方法中，默认是用CGLIB代理

1.5 声明式事务

​ 声明式事务需要在配置类中添加@EnableTransactionManagement注解

​ 该注解为容器添加两个类。AutoProxyRegistrar.class，ProxyTransactionManagementConfiguration.class

  • AutoProxyRegistrar: 为容器添加一个InfrastructureAdvisorAutoProxyCreator组件，这个组件在对象创建后，包装对象，生成对象的代理类，
  • ProxyTransactionManagementConfiguration：给容器添加事务增强器

1.6 部分接口原理

BeanFactoryPostProcessor：

他只有一个postProcessBeanFactory方法，是BeanFactory的后置处理器，beanFactory初始化后，用于修改，定制工厂的内容，他的执行时机是在beanDefiniton已经加载完成，还没实例化bean之前

2 Spring MVC基础

MVC 与三层架构的区别

  • 三层架构是针对于应用程序整体架构而言的，控制层，业务逻辑层，持久层
  • MVC是针对与应用程序的视图层划分出来的几个模块，把展现数据html页面尽量与业务代码分离，view表示展示界面，contollers层表示展示界面的业务逻辑，而Models代表views与controllers之间传递数据的一个个封装了数据的对象

两者都是为了解耦。

SpringBoot中扩展SpringMVC

在springboot中，引入了springmvc后，会自动配置一些设置，如果想要扩展，可以是继承WebMvcConfiguerAdapter来重写里面的方法，springmvc在启动时，容器中所有的WebMvcConfigurer都会一起起作用，我们的配置类也会起作用

@Configuration
public class MyMvcConfig extends WebMvcConfigurerAdapter {

    @Override
    public void addViewControllers(ViewControllerRegistry registry) {
       // super.addViewControllers(registry);
        //浏览器发送 /atguigu 请求来到 success
        registry.addViewController("/atguigu").setViewName("success");
    }
}

SpringBoot中全面接管SpringMVC

在类上加上@EnableWebMvc就可以不适用SpringmVc的默认配置，原理是@EnableWebMvc导入了DelegatingWebMvcConfiguration.class，DelegatingWebMvcConfiguration.class继承了WebMvcConfigurationSupport，而SpirngmVC的核心配置类WebMvcAutoConfiguration有这样一个条件注解@ConditionalOnMissingBean(WebMvcConfigurationSupport.class)。

3.2 springboot 缓存

CacheManager有多中不同的实现，有各种缓存技术，普遍是用k-v来存储，默认使用ConcurrentMapCacheManager

在主类上添加@EnableCaching开启注解，

4个cache方法

  1. @Cacheable 主要针对于查询操作，在查询之前判断是否有缓存，无缓存就不执行方法，反之则执行

  2. @CacheEvict 针对于删除操作，默认是删除了数据库记录后再删除缓存中的记录，方法出错那么就不删除缓存中的记录

  3. @CachePut 针对更新操作，方法执行后更新缓存，要主要key的使用。

  4. @Cacheing组合注解，用于复杂缓存策略。

缓存穿透：大量的请求缓存中不存在的数据，数据库压力就增加，从而导致数据库连接异常

缓存雪崩：缓存有过期时间，存在同一时间大面积的缓存失效，这时候大量的请求直接访问数据库。

    @Configuration
public class MyRedisConfig {
    @Bean
    public RedisTemplate<Object, Employee> redisTemplate(
            RedisConnectionFactory redisConnectionFactory)
            throws UnknownHostException {
        RedisTemplate<Object, Employee> template = new RedisTemplate<Object, Employee>();
        template.setConnectionFactory(redisConnectionFactory);
        Jackson2JsonRedisSerializer<Employee> jackson2JsonRedisSerializer=new Jackson2JsonRedisSerializer<Employee>(Employee.class);
        template.setDefaultSerializer(jackson2JsonRedisSerializer);
        return template;
    }

    @Bean
    public RedisCacheManager redisCacheManager(RedisTemplate<Object,Employee> redisTemplate){
        RedisCacheManager redisCacheManager=new RedisCacheManager(redisTemplate);
        redisCacheManager.setUsePrefix(true);
        return redisCacheManager;
    }
}

3.3 SpringBoot 消息队列

​ 消息队列是消息服务中间件，用于提升系统的异步通信能力，扩展解耦能力。

​ 两个概念：

  1. 消息代理：消息发送者发送消息后，就由消息代理接管

  2. 目的地： 消息代理保证消息传递到指定的目的地。

消息队列是异步的，有两种形式

  1. 队列：点对点式，只能有一个消息接收者，但可以多个消息接收者同时监听一个队列

  2. 主题式：发布/订阅模式，有多个消息接收者监听这个订阅信息

3.3.2 两种消息队列

JMS与AMQP对比

​ 一般使用AMQP的实现RabbitMQ。

3.3.3 RabbitMQ的消息组件

Message
消息，消息是不具名的，它由消息头和消息体组成。消息体是不透明的，而消息头则由一系列的可选属性组成，这些属性包括routing-key（路由键）、priority（相对于其他消息的优先权）、delivery-mode（指出该消息可能需要持久性存储）等。

Publisher
消息的生产者，也是一个向交换器发布消息的客户端应用程序。

Exchange
交换器，用来接收生产者发送的消息并将这些消息路由给服务器中的队列。
Exchange有4种类型：direct(默认)，fanout, topic, 和headers，不同类型的Exchange转发消息的策略有所区别

Queue
消息队列，用来保存消息直到发送给消费者。它是消息的容器，也是消息的终点。一个消息可投入一个或多个队列。消息一直在队列里面，等待消费者连接到这个队列将其取走。

Binding
绑定，用于消息队列和交换器之间的关联。一个绑定就是基于路由键将交换器和消息队列连接起来的路由规则，所以可以将交换器理解成一个由绑定构成的路由表。
Exchange 和Queue的绑定可以是多对多的关系。

Connection
网络连接，比如一个TCP连接。

Channel
信道，多路复用连接中的一条独立的双向数据流通道。信道是建立在真实的TCP连接内的虚拟连接，AMQP 命令都是通过信道发出去的，不管是发布消息、订阅队列还是接收消息，这些动作都是通过信道完成。因为对于操作系统来说建立和销毁 TCP 都是非常昂贵的开销，所以引入了信道的概念，以复用一条 TCP 连接。

Consumer
消息的消费者，表示一个从消息队列中取得消息的客户端应用程序。

Virtual Host
虚拟主机，表示一批交换器、消息队列和相关对象。虚拟主机是共享相同的身份认证和加密环境的独立服务器域。每个 vhost 本质上就是一个 mini 版的 RabbitMQ 服务器，拥有自己的队列、交换器、绑定和权限机制。vhost 是 AMQP 概念的基础，必须在连接时指定，RabbitMQ 默认的 vhost 是 / 。

Broker
表示消息队列服务器实体

三种消息模型：

  1. direct：精确匹配，route key（路由键：用于exchange中判断消息进入哪个队列）与队列名精确匹配
  2. fanout：route key 无效，消息进入fanout的exchange中将会进入到所有绑定的队列中
  3. Topic：route key使用模糊匹配 ，# 匹配0到1个单词，*匹配1个单词。

3.4 springboot 检索

它的组件概念有些歧义，具体看下边。

4. Spring项目中遇到的问题

4.1 对象之间属性的复制

业务场景：更新用户，局部更新用户的某些属性，因为只传过来部分属性，Spring data jpa 的save方法当属性为空时，更新时，不会保留数据库中的原有属性，所有就要进行属性拷贝，先查询到数据库的对象，然后与更新的属性进行整合。

步骤： 引入依赖

<dependency>
    <groupId>cn.hutool</groupId>
    <artifactId>hutool-all</artifactId>
    <version>4.1.14</version>
</dependency>

利用方法：

BeanUtil.copyProperties(oldObject,newObject,true, CopyOptions.create().setXXXX(true))

将oldobject的属性拷贝到newObject中，setXXX可设置选项

  • editable 限制的类或接口，必须为目标对象的实现接口或父类，用于限制拷贝的属性，例如一个类我只想复制其父类的一些属性，就可以将editable设置为父类。

  • ignoreNullValue 是否忽略空值，当源对象的值为null时，true: 忽略而不注入此值，false: 注入null

  • ignoreProperties 忽略的属性列表，设置一个属性列表，不拷贝这些属性值
    ignoreError 是否忽略字段注入错误

具体代码：

public Integer update(User user) {
        User origin = userDao.findOne(user.getId());
        //将新的user 的非空属性传入旧的user
        BeanUtil.copyProperties(user,origin,true, CopyOptions.create().setIgnoreNullValue(true));
        System.out.println(user.getTrueName());

        //把没有值的数据  换成原数据库的数据。
//        user = repalce(user, origin);

        userDao.save(origin);
        return 1;
    }
